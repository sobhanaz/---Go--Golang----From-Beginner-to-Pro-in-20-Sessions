# جلسه‌ی ۲۰ — جلا و انتشار 🚢

**هدف (۱ ساعت):** TaskFlow را از «روی دستگاه من کار می‌کند» به «یک کارفرما می‌تواند
کلون و اجرا کند و تحت‌تأثیر قرار بگیرد» برسانی. **خاموشی برازنده (graceful
shutdown)**، **Docker**، یک **Makefile** و یک **README** جلاخورده اضافه می‌کنی —
بعد کل پروژه را به **بولت‌پوینت‌های رزومه** و نکات گفتگوی مصاحبه تبدیل می‌کنی. این
جلسه کارت را *قابل‌ارائه* می‌کند.

> **مرور جلسه‌ی ۱۹:** TaskFlow یک REST API امن، چندکاربره و تست‌شده است. حالا
> قابل‌استقرار و قابل‌ارائه‌اش می‌کنیم — فرق یک اسکریپت سرگرمی با یک نمونه‌کار.

> 📂 همه‌ی کد در [`taskflow/`](../taskflow/) است.

## ۱. خاموشی برازنده (۱۵ دقیقه)

یک سرور ساده (`http.ListenAndServe(...)`) هنگام کشته‌شدن فوراً می‌میرد و
درخواست‌های در حال اجرا را رها می‌کند. سرورهای محصول **برازنده خاموش می‌شوند**:
پذیرش درخواست جدید را متوقف، درخواست‌های فعلی را تمام، بعد خارج می‌شوند. به
[`taskflow/main.go`](../taskflow/main.go) نگاه کن:

</div>

```go
srv := &http.Server{Addr: cfg.Addr, Handler: server.Routes(), /* timeoutها */}

// سرور را در گوروتین اجرا کن تا main منتظر سیگنال بماند.
go func() {
    if err := srv.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
        log.Fatalf("server error: %v", err)
    }
}()

// تا رسیدن سیگنال interrupt/termination مسدود شو.
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit

// به درخواست‌های در حال اجرا تا ۱۰ ثانیه فرصت اتمام بده.
ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()
srv.Shutdown(ctx)
```

<div dir="rtl">

این همه‌ی آنچه یاد گرفتی را ترکیب می‌کند: **گوروتین** (جلسه‌ی ۱۲)، **کانال** برای
انتظار سیگنال، و **context** با مهلت (جلسه‌ی ۱۳). `http.Server` همچنین
`ReadTimeout`/`WriteTimeout`/`IdleTimeout` تنظیم می‌کند — سخت‌سازی پایه تا
کلاینت‌های کند منابع را اشغال نکنند.

> 🔑 **چرا مهم است:** ارکستریتورهایی مثل کوبرنتیز قبل از توقف کانتینر `SIGTERM`
> می‌فرستند. یک سرور برازنده کارش را تمام و تمیز خارج می‌شود نه این که وسط درخواست
> خطا بدهد. مصاحبه‌گرها عاشق دیدن این‌اند.

## ۲. Docker‌ای کردن — یک ایمیج کوچک و امن (۲۰ دقیقه)

یک Dockerfile **چندمرحله‌ای** در یک ایمیج بیلد می‌کند و در یک ایمیج مینیمال منتشر
می‌کند. به [`taskflow/Dockerfile`](../taskflow/Dockerfile) نگاه کن:

</div>

```dockerfile
# ---- مرحله‌ی بیلد ----
FROM golang:1.25 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download              # تا وابستگی‌ها عوض نشوند کش می‌شود
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /taskflow .

# ---- مرحله‌ی اجرا ----
FROM gcr.io/distroless/static-debian12
COPY --from=build /taskflow /app/taskflow
EXPOSE 8080
ENTRYPOINT ["/app/taskflow"]
```

<div dir="rtl">

سه ایده که این را *خوب* می‌کنند، نه فقط کارکن:

۱. **بیلد چندمرحله‌ای.** زنجیره‌ابزار سنگین Go در مرحله‌ی بیلد می‌ماند؛ ایمیج نهایی
   فقط فایل اجرایی تو را دارد. نتیجه: ایمیج ~۱۵ مگابایتی به‌جای ~۱ گیگابایتی.
۲. **`CGO_ENABLED=0`** یک باینری کاملاً ایستا تولید می‌کند. این فقط ممکن است چون
   TaskFlow از درایور **خالص-Go** اسکوئل‌لایت (`modernc.org/sqlite`) استفاده می‌کند
   — یک انتخاب عمدی از جلسه‌ی ۱۸ که حالا ثمر می‌دهد.
۳. **پایه‌ی distroless.** `distroless/static` نه شل دارد نه پکیج‌منیجر، پس سطح حمله
   کوچک است. `-ldflags="-s -w"` اطلاعات دیباگ را حذف می‌کند تا باینری کوچک‌تر شود.

</div>

```bash
docker build -t taskflow .
docker run --rm -p 8080:8080 -e TASKFLOW_JWT_SECRET=change-me taskflow
```

<div dir="rtl">

یک [`.dockerignore`](../taskflow/.dockerignore) زمینه‌ی بیلد را کوچک نگه می‌دارد، و
یک [`.gitignore`](../taskflow/.gitignore) باینری و `*.db` را از گیت بیرون نگه می‌دارد.

## ۳. ارگونومی توسعه‌دهنده: Makefile (۱۰ دقیقه)

یک [`Makefile`](../taskflow/Makefile) به هرکسی ورودی یک‌کلمه‌ای به کارهای رایج
می‌دهد — لمسی کوچک که نشانه‌ی دقت است:

</div>

```bash
make help          # فهرست دستورها
make run           # go run .
make test          # go test ./...
make cover         # go test -cover ./...
make docker-build  # ساخت ایمیج
```

<div dir="rtl">

> 💡 به Make *نیاز* نداری، اما `make test` / `make run` دوستانه‌تر از به‌خاطرسپردن
> فلگ‌هاست، و کارفرماها می‌بینند که به توسعه‌دهنده‌ی بعدی فکر می‌کنی.

## ۴. README‌ای که برایت مصاحبه می‌آورد (۱۰ دقیقه)

برای یک پروژه‌ی نمونه‌کار، **README مهم‌ترین فایل است** — اولین (اغلب تنها) چیزی که
کارفرما می‌خواند. یک README قوی به‌ترتیب دارد:

۱. **توضیح یک‌خطی** این که چیست.
۲. **بولت‌های ویژگی** — چه می‌کند، با کلیدواژه‌های مهم (JWT، REST، Docker، تست).
۳. جدول **پشته‌ی فناوری**.
۴. **معماری** — درخت پوشه + یک پاراگراف درباره‌ی طراحی.
۵. **دستورهای اجرا** — محلی *و* Docker، قابل کپی‌پیست.
۶. **مرجع API** — endpointها، با یک مثال `curl` کارشده.
۷. **نحوه‌ی تست.**

فایل [`taskflow/README.md`](../taskflow/README.md) دقیقاً همین‌طور نوشته شده. به‌عنوان
الگو مطالعه‌اش کن؛ یک README واضح اغلب بیش از خود کد تحت‌تأثیر می‌گذارد، چون بیشتر
بازبین‌ها فقط مرور می‌کنند.

## ۵. گذاشتن TaskFlow در رزومه (۵ دقیقه)

پروژه را با **تأثیر و جزئیات** توصیف کن، نه فقط «یک API ساختم». چند بولت مثل این‌ها
برای رزومه‌ات انتخاب کن:

> **TaskFlow — REST API (Go)** · [github.com/you/taskflow]
> - یک REST API مدیریت وظایف چندکاربره در **Go** با کتابخانه‌ی استاندارد
>   `net/http`، با **احراز هویت JWT** و هش رمز bcrypt ساختم.
> - یک **معماری تمیز و لایه‌ای** (هندلر ← مخزن ← دیتابیس) طراحی کردم که با اینترفیس
>   جدا شده و امکان تست هندلرها در برابر یک دیتابیس واقعی را می‌دهد.
> - **میان‌افزار** برای لاگ ساختاریافته، بازیابی از panic و احراز هویت، و
>   **ایزولاسیون داده به‌ازای کاربر** اعمال‌شده در لایه‌ی SQL پیاده کردم.
> - **تست‌های یکپارچگی** با `net/http/httptest` نوشتم که احراز هویت، CRUD و
>   کنترل دسترسی را پوشش می‌دهد.
> - با یک بیلد **Docker چندمرحله‌ای** که یک ایمیج ایستای distroless ~۱۵ مگابایتی
>   تولید می‌کند **کانتینری‌اش کردم**، با خاموشی برازنده برای استقرار بدون‌خاموشی.

### نکات گفتگوی مصاحبه (آماده باش *چرا* را توضیح دهی)

- **«چرا بدون فریم‌ورک؟»** — کتابخانه‌ی استاندارد Go برای یک REST API تمیز کافی
  است؛ وابستگی کمتر، استدلال آسان‌تر.
- **«احراز هویت چطور کار می‌کند؟»** — JWTهای بدون‌حالت: توکن امضاشده شناسه‌ی کاربر
  را حمل می‌کند، پس سرور به‌جای ذخیره‌ی نشست، امضا را تأیید می‌کند.
- **«وظایف چطور خصوصی می‌مانند؟»** — هر کوئری با `WHERE user_id = ?` محدود می‌شود؛
  یک کاربر فیزیکی نمی‌تواند ردیف‌های دیگری را بخواند.
- **«چطور مقیاسش می‌دهی؟»** — اسکوئل‌لایت را با پستگرس پشت همان اینترفیس
  `TaskRepository` عوض کن؛ هندلرها عوض نمی‌شوند.
- **«هندلرها را چطور تست می‌کنی؟»** — `httptest` بدون شبکه‌ی واقعی به روتر درخواست
  می‌فرستد، در برابر یک دیتابیس موقت — سریع و قطعی.

## 🎯 تمرین‌های پایانی — مالِ *خودت* کن

۱. **به گیت‌هاب منتشرش کن.** یک repo بساز، commit و کل دوره‌ی `golang/` (یا فقط `taskflow/`) را push کن. تاریخچه‌ی commit سبز خودش یک سیگنال است.
۲. **یک ویژگی** سرتاسری اضافه کن و تستش را بنویس: مثلاً **اولویت** وظیفه، **تاریخ سررسید**، یا فیلتر `GET /tasks?done=true`. به هر لایه دست بزن (مدل ← store ← هندلر ← تست).
۳. **ایمیج Docker را بساز و اجرا کن** به‌صورت محلی؛ تأیید کن API در کانتینر کار می‌کند.
۴. **بولت‌های رزومه‌ات** را برای TaskFlow با الگوی بالا بنویس، متناسب با ویژگی‌ای که *تو* اضافه کردی.
۵. **به یادگیری ادامه بده:** `log/slog` (لاگ ساختاریافته) را کاوش کن، یک workflow برای CI اضافه کن که `go test ./...` را اجرا کند، یا پستگرس را جایگزین کن.

## 🎓 دوره را تمام کردی!

از `package main` تا یک REST API کانتینری، احرازهویت‌شده و تست‌شده رسیدی. حالا نحو
Go، ساختمان‌داده‌ها، اینترفیس، خطا، همروندی، کتابخانه‌ی استاندارد، تست و HTTP را
می‌دانی — و یک **پروژه‌ی واقعی** برای اثباتش داری. این مجموعه‌مهارتِ یک توسعه‌دهنده‌ی
بک‌اند Go در سطح جونیور-تا-میانی است.

**بعد کجا برویم:** یک پروژه‌ی کوچک *دیگر* را از صفر بساز (کوتاه‌کننده‌ی URL، API
ردیاب هزینه) بدون نگاه به یادداشت‌ها — این‌طور به حافظه‌ی عضلانی تبدیل می‌شود. بعد
کد واقعی Go بخوان: کتابخانه‌ی استاندارد، و پروژه‌هایی مثل `chi`، `sqlc`، یا منبع
ابزارهایی که استفاده می‌کنی. به جامعه‌ی Go خوش آمدی. 🐹
